## Problem Statement

### Small Business Operations Platform

Small businesses with around 10 to 50 employees commonly run operations using fragmented tools such as spreadsheets, chat apps, email threads, and ad-hoc scripts. This causes low scalability, weak auditability, and operational friction as business volume increases.

Objective:
Design and implement a structured, scalable, and maintainable software platform for internal operational workflows, with incremental delivery and explicit AI-assisted software engineering practices.

## Scope Expectations

Each team defines:
- Type of small business addressed
- Primary pain points
- Key workflows
- User roles involved

Examples of target businesses:
- Retail store
- Restaurant or cafe
- Small manufacturing unit
- Logistics or delivery service
- Professional services firm
- Local healthcare clinic (non-clinical operations)
- Training or coaching institute

## Functional Expectations

The system must:
- Support at least two distinct user roles
- Manage interrelated workflows such as:
  - Orders/jobs/requests
  - Inventory/resources
  - Scheduling/assignments
  - Payments/invoices (without real payment processing)
  - Customer/vendor interactions
- Expose well-defined APIs for core functionality
- Include automated tests for critical business logic
- Be designed for growth and requirement change

## AI Usage Requirement

Every submission must include:
- How AI was used
- Prompt history/conversation URL(s)
- Any agentic AI tools used
- Reflection on benefits and challenges of AI-assisted development

## Milestone 1 Deliverables

Focus: User requirements discovery

Required outputs:
- User identification (primary, secondary, tertiary)
- Interview/observation evidence focused on pain points
- SMART user stories in required format:
  - As a [type of user]
  - I want [an action]
  - So that [a benefit/value]

Peer evaluation focus:
- User identification quality
- Pain-point discovery quality
- SMART compliance of user stories

## Milestone 2 Deliverables

Focus: Scheduling and design

Required outputs:
- Project and sprint schedules (including scrum cadence)
- Scheduling tool evidence (Jira/Trello/Gantt/Kanban)
- Component design
- Class diagrams
- Database design diagram
- Scrum minutes
- Frontend page screenshots and runnable frontend package

Peer evaluation focus:
- Completeness of planning artifacts
- Design quality and coverage
- Scrum documentation
- UI page breadth

## Sprint 1 and Sprint 2 Deliverables

Focus: APIs, tests, and user testing

Required outputs:
- API list (created and integrated)
- API descriptions mapped to user stories
- Swagger-compatible YAML for team-created APIs
- Backend implementation code
- Extensive API test cases in format:
  - API being tested
  - Inputs
  - Expected output
  - Actual output
  - Result (Success/Fail)
- pytest code for implemented/updated APIs
- User feedback and sprint planning updates

Sprint 2 adds:
- Updated YAML and code based on feedback
- Updated API test cases and pytest results

Peer evaluation focus:
- API-YAML completeness and user-story mapping
- Code quality and implementation fidelity
- Test depth and formatting quality
- pytest usage and defect highlighting
